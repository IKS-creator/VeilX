# VeilX — Спецификация продукта

> Приватный VPN-сервис для узкого круга (семья + друзья).
> Домен: veilx.app | Сервер: Vultr Stockholm | Протокол: VLESS + Reality

---

## 1. ПРОБЛЕМА И РЕШЕНИЕ

### Проблема

Пользователи в России сталкиваются с тройной болью:

1. **Блокировки сервисов** — Instagram, YouTube, ChatGPT и десятки других ресурсов недоступны без VPN.
2. **Приватность** — провайдеры видят весь трафик, DPI-системы анализируют содержимое.
3. **Коммерческие VPN ненадёжны** — блокируются РКН (протоколы OpenVPN/WireGuard детектируются), тормозят на перегруженных серверах, стоят $5-12/мес на человека.

Для группы из 20 человек коммерческий VPN = $100-240/мес. При этом качество нестабильно.

### Решение

Собственный VPS в Стокгольме ($6/мес на всех) с протоколом VLESS + Reality, который маскируется под обычный HTTPS-трафик и не детектируется DPI/РКН.

- Админ создаёт инвайт-ссылку и отправляет другу
- Друг открывает ссылку — видит свой конфиг, QR-код и инструкции
- Скачивает клиент (FoXray / v2rayNG) → сканирует QR → подключён

### Метрика успеха

- Было: $5-12/мес на человека, нестабильное соединение, блокировки
- Стало: $0.35/мес на человека ($7/мес на 20 юзеров), стабильное соединение, протокол не блокируется

## 2. ПОЛЬЗОВАТЕЛИ И РОЛИ

### Роли

| Роль | Кол-во | Описание |
|------|--------|----------|
| Админ | 1 | Владелец сервиса. Полный доступ ко всему. |
| Пользователь | 10-20 | Семья и друзья. Доступ только к своей персональной странице. |

### Авторизация

- **Админ**: вход в админ-панель `/admin` по паролю (единый пароль, хранится в env-переменной `ADMIN_PASSWORD` на Vercel). Пароль сравнивается plain-text (constant-time comparison через `crypto.timingSafeEqual`). После успешного входа — редирект на панель. При истечении сессии — редирект на форму входа.
- **Защита от bruteforce**: максимум 5 попыток входа за 15 минут с одного IP. После превышения — ответ 429 «Слишком много попыток, попробуй позже». Счётчик хранится в Vercel Postgres (таблица `login_attempts`: ip VARCHAR, attempts INT, window_start TIMESTAMP). На serverless Map не работает — каждый cold start обнуляет память.
- **Пользователь**: без регистрации и логина. Доступ по уникальному секретному URL `veilx.app/c/{token}`, где `token` — случайная строка 16 символов (a-z, 0-9), генерируется через `BigInt('0x' + crypto.randomBytes(12).toString('hex')).toString(36).slice(0, 16)`. URL является единственным фактором авторизации.

### Как попасть в систему

1. Админ генерирует инвайт в админ-панели (вводит имя друга).
2. Система создаёт UUID для VLESS, уникальный token для URL, записывает в БД.
3. Админ копирует ссылку `veilx.app/c/{token}` и отправляет другу (Telegram, WhatsApp, лично).
4. Друг переходит по ссылке — видит конфиг. Регистрация не требуется.

### 3.3 Персональная страница (`/c/{token}`)

**Назначение**: страница конкретного пользователя с его конфигом.

**Как попасть**: по уникальной ссылке от админа.

**Элементы:**
- Приветствие: «Привет, {name}!»
- Блок конфига:
  - VLESS-ссылка (текстовое поле, замаскирована точками `•••••••••` по умолчанию, кнопка «Показать» / «Скрыть» toggle)
  - Кнопка «Скопировать конфиг» — копирует vless:// ссылку в буфер обмена через Clipboard API, текст кнопки меняется на «Скопировано ✓» на 2 секунды, затем возвращается
  - QR-код — отрендеренный из vless:// ссылки библиотекой `qrcode.react`, размер 280×280px, error correction level M, чёрный на белом
- Ссылка «Как подключиться?» → ведёт на `/setup`

**Формат VLESS-ссылки:**
```
vless://{uuid}@{VPS_IP}:443?encryption=none&flow=xtls-rprx-vision&security=reality&sni={REALITY_SNI}&fp=chrome&pbk={REALITY_PUBLIC_KEY}&sid={REALITY_SHORT_ID}&type=tcp#{name}
```
Где:
- `{uuid}` — UUID пользователя из БД
- `{VPS_IP}` — IP VPS-сервера (env `VPS_IP`)
- `{REALITY_SNI}` — домен маскировки (env `REALITY_SNI`, по умолчанию `dl.google.com`)
- `{REALITY_PUBLIC_KEY}` — публичный ключ Reality (env `REALITY_PUBLIC_KEY`)
- `{REALITY_SHORT_ID}` — короткий ID Reality, 8 hex-символов (env `REALITY_SHORT_ID`)
- `{name}` — имя пользователя (URL-encoded), отображается как название конфига в клиенте
- Ссылка формируется **на сервере** (в API route / SSR), чтобы не раскрывать env-переменные клиенту

**Состояния:**

| Состояние | Что видит пользователь |
|-----------|----------------------|
| Активен | Конфиг, QR-код, кнопка копирования |
| Отключён | Сообщение: «Твой доступ приостановлен. Свяжись с админом.» |
| Неверный token | Страница 404: «Страница не найдена» |

### 3.4 Админ-панель (`/admin`)

**Назначение**: управление пользователями и инвайтами.

**Как попасть**: URL `/admin`, ввод пароля.

**Элементы:**

**Форма входа** (если не авторизован):
- Поле пароля
- Кнопка «Войти»
- При неверном пароле: сообщение «Неверный пароль» (красный текст)
- Авторизация: подписанный JWT в httpOnly secure cookie с именем `veilx-admin`. Payload: `{ role: "admin", iat }`. Секрет подписи: env `JWT_SECRET`. Срок жизни: 24 часа (один админ — короткая сессия безопаснее). При истечении — редирект на форму входа с сообщением «Сессия истекла, войди снова».

**Панель** (после входа):

- **Кнопка «Синхронизировать с сервером»** — POST /api/admin/sync, показывает спиннер, при успехе — зелёный toast «Синхронизировано», при ошибке — красный toast с текстом ошибки

- **Блок «Создать инвайт»** (скрыт если ≥ 20 юзеров, вместо него текст о лимите):
  - Поле ввода: «Имя» (обязательное, текст, до 50 символов, trim при submit)
  - Кнопка «Создать» (disabled пока поле пустое или в процессе запроса)
  - После создания: inline-блок под формой (не модалка) с ссылкой `veilx.app/c/{token}` и кнопкой «Скопировать». Блок исчезает при создании следующего инвайта.

- **Таблица пользователей**:
  - Столбцы: Имя | Статус | Трафик (↑/↓) | Последнее подключение | Действия
  - Сортировка: по умолчанию по `created_at` DESC (новые сверху), клик по заголовку не меняет сортировку (при 20 юзерах не нужно)
  - Статус: зелёный кружок «Активен» / серый «Отключён»
  - Трафик: отображается в human-readable формате. Правила: < 1 МБ → «0 МБ», 1 МБ–1 ГБ → «X МБ», ≥ 1 ГБ → «X.X ГБ» (одна десятичная). Вверх и вниз показываются через `/`: «150 МБ / 2.3 ГБ»
  - Последнее подключение: дата и время в формате `DD.MM.YYYY HH:mm` в UTC+3 (Moscow), или «Никогда» если `last_connected_at` = NULL
  - Действия:
    - «Скопировать ссылку» — копирует персональный URL, feedback «Скопировано ✓» на 2 сек
    - «Отключить» / «Включить» — переключает статус. Кнопка показывает спиннер на время запроса, при ошибке — toast «Не удалось выполнить действие»
    - «Удалить» — модальное окно (не browser confirm): «Точно удалить {name}? Это действие необратимо.» с кнопками «Удалить» (красная) и «Отмена»

**Состояния:**

| Состояние | Что видит админ |
|-----------|----------------|
| Нет пользователей | Таблица с текстом «Пока никого нет. Создай первый инвайт.» |
| Есть пользователи | Полная таблица со всеми данными |
| Загрузка | Skeleton-загрузка в таблице (серые полоски на месте строк) |
| Ошибка связи с VPS | Жёлтый баннер над таблицей: «⚠ Не удалось получить статистику с сервера. Данные о трафике могут быть неактуальны.» Баннер не блокирует действия. |
| Лимит 20 юзеров достигнут | Форма создания инвайта заблокирована, вместо неё текст: «Достигнут лимит в 20 пользователей. Удали неактивных, чтобы создать новых.» |
| Действие в процессе | Кнопка действия показывает спиннер, остальные кнопки строки disabled |

## 4. ДЕЙСТВИЯ ПОЛЬЗОВАТЕЛЯ (User Flows)

### Flow 1: Админ создаёт инвайт

```
Триггер: Админ хочет дать VPN другу.

1. Админ открывает /admin
2. Вводит пароль (если не залогинен) → попадает в панель
3. В поле «Имя» вводит имя друга (например, «Маша»)
4. Нажимает «Создать»
5. Система:
   a. Генерирует UUID v4 для VLESS
   b. Генерирует token (16 символов, a-z0-9)
   c. Формирует vless:// ссылку с UUID, адресом сервера, параметрами Reality
   d. Записывает в Vercel Postgres: name, token, uuid, status=active, created_at
   e. Отправляет запрос к VPS API: добавить UUID в конфиг Xray → reload Xray
6. Система показывает ссылку veilx.app/c/{token}
7. Админ копирует ссылку, отправляет другу в Telegram

Результат: Друг может подключиться.
```

**Ошибки:**
- Имя пустое → кнопка «Создать» неактивна (disabled + серая)
- Имя содержит только пробелы → trim перед валидацией, если после trim пусто — считается пустым
- Имя-дубликат → допускается (разные люди могут иметь одинаковые имена, уникальность — по token/uuid)
- Достигнут лимит 20 юзеров → форма заблокирована (см. состояния админ-панели)
- VPS недоступен (таймаут 10 сек) → инвайт создаётся в БД со статусом `active`, жёлтый баннер «Сервер временно недоступен, конфиг будет активирован при восстановлении связи». UUID будет добавлен в Xray при следующем успешном действии или ручной синхронизации.
- Кнопка «Создать» показывает спиннер на время запроса и disabled, чтобы предотвратить двойное нажатие

---

### Flow 2: Пользователь подключается к VPN

```
Триггер: Пользователь получил ссылку от админа.

1. Открывает ссылку veilx.app/c/{token} в браузере
2. Видит: приветствие, конфиг (скрыт), кнопку «Скопировать», QR-код, ссылку на инструкции
3a. На телефоне: сканирует QR-код камерой клиента (FoXray / v2rayNG)
3b. На компьютере: нажимает «Скопировать конфиг» → открывает клиент → вставляет
4. В клиенте нажимает «Подключиться»
5. VPN подключён. Проверка: ip.me показывает IP Стокгольма.

Результат: Пользователь онлайн через VPN.
```

**Ошибки:**
- Неверный token → страница 404
- Статус «Отключён» → сообщение «Доступ приостановлен»

---

### Flow 3: Админ отключает пользователя

```
Триггер: Админ хочет отключить доступ.

1. Админ открывает /admin
2. Находит пользователя в таблице
3. Нажимает «Отключить»
4. Система:
   a. Обновляет status → disabled в Vercel Postgres
   b. Отправляет запрос к VPS API: удалить UUID из конфига Xray → reload Xray
5. Строка в таблице обновляется: статус серый «Отключён», кнопка меняется на «Включить»

Результат: VPN-соединение пользователя прерывается, персональная страница показывает «Доступ приостановлен».
```

---

### Flow 4: Админ удаляет пользователя

```
Триггер: Админ хочет полностью удалить пользователя.

1. Админ нажимает «Удалить» в строке пользователя
2. Появляется подтверждение: «Точно удалить {name}? Это действие необратимо.»
3. Админ подтверждает
4. Система:
   a. Удаляет UUID из конфига Xray на VPS → reload
   b. Удаляет запись из Vercel Postgres
5. Строка исчезает из таблицы

Результат: Пользователь полностью удалён, ссылка возвращает 404.
```

---

### Flow 5: Админ включает пользователя обратно

```
Триггер: Админ хочет вернуть доступ отключённому пользователю.

1. Админ нажимает «Включить» в строке отключённого пользователя
2. Система:
   a. Обновляет status → active в Vercel Postgres
   b. Отправляет запрос к VPS API: добавить UUID в конфиг Xray → reload
3. Строка обновляется: статус зелёный «Активен»

Результат: VPN снова работает, персональная страница показывает конфиг.
```

## 5. БИЗНЕС-ЛОГИКА И ПРАВИЛА

| Правило | Значение |
|---------|----------|
| Макс. пользователей | 20 (мягкий лимит, ограничен ресурсами VPS) |
| Лимит трафика | Нет. Общий лимит VPS — 2 ТБ/мес от Vultr |
| Лимит устройств на конфиг | Нет. Один конфиг можно использовать на нескольких устройствах |
| Лимит скорости | Нет. Скорость определяется каналом VPS (1 Gbps у Vultr) |
| Длина token | 16 символов, a-z0-9 (36^16 ≈ 7.96 × 10^24 комбинаций — bruteforce нереален) |
| UUID формат | UUID v4 для VLESS |
| Сессия админа | JWT в httpOnly secure cookie `veilx-admin`, срок 24 часа, секрет `JWT_SECRET` |
| Пароль админа | Хранится в `ADMIN_PASSWORD` на Vercel, сравнивается через `crypto.timingSafeEqual` |
| Rate limiting | 5 попыток входа / 15 мин с одного IP, потом 429. Счётчик в Postgres (serverless-совместимо) |
| Протокол VPN | VLESS + Reality (Raw), flow: `xtls-rprx-vision`, fp: `chrome`, SNI: `dl.google.com` (env `REALITY_SNI`) |

## 6. ДАННЫЕ

### Таблица: users

| Поле | Тип | Описание |
|------|-----|----------|
| id | serial, PK | Автоинкремент |
| name | varchar(50), NOT NULL | Имя пользователя (для идентификации админом) |
| token | varchar(16), UNIQUE, NOT NULL | Секретный токен для персонального URL |
| vless_uuid | uuid, UNIQUE, NOT NULL | UUID для VLESS-конфига |
| status | varchar(10), NOT NULL, DEFAULT 'active' | 'active' или 'disabled' |
| traffic_up | bigint, DEFAULT 0 | Исходящий трафик в байтах |
| traffic_down | bigint, DEFAULT 0 | Входящий трафик в байтах |
| last_connected_at | timestamp, NULL | Время последнего подключения |
| created_at | timestamp, DEFAULT now() | Дата создания |
| updated_at | timestamp, DEFAULT now() | Дата последнего изменения (обновляется при смене status) |

### Таблица: login_attempts (rate limiting)

| Поле | Тип | Описание |
|------|-----|----------|
| ip | varchar(45), PK | IP-адрес (IPv4 или IPv6) |
| attempts | int, DEFAULT 0 | Кол-во попыток |
| window_start | timestamp, DEFAULT now() | Начало окна 15 минут |

> Очистка: записи старше 15 минут удаляются при каждой проверке (self-cleaning).

### Индексы
- `users_token_idx` UNIQUE на `token` (поиск по персональному URL)
- `users_vless_uuid_idx` UNIQUE на `vless_uuid`

### Каскадное поведение
- При удалении пользователя: запись удаляется из Postgres + UUID удаляется из Xray.
- Внешних связей нет (одна таблица).

## 7. ФОНОВЫЕ ПРОЦЕССЫ

### 7.1 Сбор статистики (cron, раз в час)

```
Запуск: Vercel Cron Job, раз в сутки (0 3 * * *) — Hobby plan поддерживает только daily cron.
  Для почасовой статистики нужен Pro plan ($20/мес). MVP — ежедневно.
Аутентификация: Vercel передаёт заголовок Authorization: Bearer {CRON_SECRET}. 
  API route проверяет CRON_SECRET перед выполнением. Без валидного токена — 401.
Действие:
  1. Отправляет запрос к VPS API: GET /stats (таймаут 30 сек)
  2. VPS API вызывает xray api statsquery --reset (gRPC) — получает дельту трафика с прошлого вызова
  3. Vercel API route получает ответ в формате:
     { "users": { "{uuid}": { "up": 12345, "down": 67890, "online": true } } }
  4. Обновляет в Postgres:
     - traffic_up += delta_up, traffic_down += delta_down (аккумулятивно)
     - last_connected_at = now() если online=true
Важно: --reset обнуляет счётчики Xray после чтения. При перезапуске Xray счётчики 
  тоже обнуляются — незаписанная дельта теряется (допустимая потеря: до 1 часа данных).
Сбой:
  - VPS недоступен → логируется console.error, данные не обновляются, следующий запуск попробует снова
  - Не является критическим: админка показывает баннер «Данные могут быть неактуальны»
```

### 7.2 Автоперезапуск Xray (systemd)

```
Конфигурация: systemd unit xray.service с Restart=on-failure
Действие: при падении Xray — автоматический перезапуск через 5 секунд
Лимит: 3 перезапуска за 60 секунд, потом остановка (чтобы не зациклить)
```

### 7.3 Синхронизация конфига Xray при изменениях

```
Триггер: создание, отключение, включение, удаление пользователя
Таймаут: 10 секунд на запрос к VPS API. Retry: нет (fail fast, sync потом).
Действие:
  1. Next.js API route отправляет команду к VPS API (POST/DELETE)
  2. VPS API модифицирует /usr/local/etc/xray/config.json (добавляет/удаляет UUID в clients[])
  3. VPS API выполняет: `pidof xray && kill -SIGHUP $(pidof xray) || systemctl restart xray` — reload без разрыва соединений, fallback на restart если процесс не найден
Сбой:
  - VPS недоступен (таймаут или ошибка) → изменение сохранено в БД, жёлтый баннер админу
  - Кнопка «Синхронизировать с сервером» в админке (видна всегда): 
    POST /api/admin/sync → отправляет ТОЛЬКО active UUID из БД → VPS API /sync → полная перезапись clients[] (disabled UUID исключаются из Xray)
```

## 9. ПОРЯДОК СБОРКИ

### Фаза 1: VPN-сервер (VPS)

**Зависимости**: нет, можно начинать сразу.

1. Подключиться к VPS по SSH (root@70.34.204.16)
2. Обновить систему: `apt update && apt upgrade`
3. Установить Xray-core (последняя версия с GitHub)
4. Настроить конфиг Xray:
   - Протокол: VLESS
   - Транспорт: Raw (TCP)
   - Security: Reality
   - SNI: `dl.google.com` (проверить: `curl -I https://dl.google.com` с IP VPS должен дать ответ — домен должен быть доступен)
   - Создать пару ключей Reality: `xray x25519` → записать private key в конфиг Xray, public key → env `REALITY_PUBLIC_KEY`
   - Сгенерировать Short ID: `openssl rand -hex 4` → записать в конфиг Xray и env `REALITY_SHORT_ID`
   - Порт: 443
   - Flow: `xtls-rprx-vision`
5. Настроить systemd unit для Xray (автоперезапуск)
6. Настроить файрвол (ufw): `ufw allow 22/tcp && ufw allow 443/tcp && ufw allow 8443/tcp && ufw enable`
7. Проверить подключение с клиента вручную

### Фаза 2: API на VPS

**Зависимости**: Фаза 1 (Xray должен работать).

1. Установить Node.js 20 LTS на VPS: `curl -fsSL https://deb.nodesource.com/setup_20.x | bash - && apt install -y nodejs`
2. Сгенерировать self-signed TLS-сертификат: `openssl req -x509 -newkey ec -pkeyopt ec_paramgen_curve:prime256v1 -nodes -keyout /opt/veilx-api/key.pem -out /opt/veilx-api/cert.pem -days 3650 -subj "/CN=veilx-api"`
3. Создать Express.js мини-API (`/opt/veilx-api/`), запуск через `https.createServer({cert, key}, app)`:
   - `POST /users` — тело: `{ "uuid": "..." }`. Добавить UUID в clients[] конфига Xray → SIGHUP reload → 200
   - `DELETE /users/:uuid` — удалить UUID из clients[] → SIGHUP reload → 200
   - `GET /stats` — вызвать `xray api statsquery --server=127.0.0.1:10085 --reset` → распарсить вывод → вернуть JSON `{ "users": { "uuid": { "up": N, "down": N, "online": bool } } }`
   - `POST /sync` — тело: `{ "uuids": ["...", "..."] }`. Перезаписать clients[] в конфиге → SIGHUP reload → 200
   - Все ответы: JSON. Ошибка: `{ "error": "message" }` + HTTP 4xx/5xx
4. Middleware: проверка `Authorization: Bearer {API_TOKEN}` через `crypto.timingSafeEqual`. Невалидный → 401.
5. Запустить HTTPS на порту 8443 (без nginx — упрощает архитектуру)
6. Настроить systemd unit `veilx-api.service` с `EnvironmentFile=/etc/veilx-api.env`, `Restart=on-failure`
7. Включить Xray Stats API: в конфиге Xray добавить секцию `"stats": {}` и `"api": { "services": ["StatsService"], "tag": "api" }` + inbound на `127.0.0.1:10085` с protocol `dokodemo-door`

### Фаза 3: База данных

**Зависимости**: нет, можно параллельно с Фазами 1-2.

1. Создать Vercel Postgres instance
2. Создать таблицу `users` по схеме из раздела 6
3. Настроить переменные окружения на Vercel: DATABASE_URL, ADMIN_PASSWORD, VPS_API_URL, VPS_API_TOKEN

### Фаза 4: Next.js API routes

**Зависимости**: Фазы 2 + 3 (нужны VPS API и БД).

1. Инициализировать Next.js проект с App Router
2. Подключить Vercel Postgres (@vercel/postgres)
3. Создать API routes:
   - `POST /api/admin/login` — проверка пароля, установка cookie
   - `GET /api/admin/users` — список пользователей из БД
   - `POST /api/admin/users` — создать инвайт (генерация UUID + token → БД + VPS API)
   - `PATCH /api/admin/users/{id}` — вкл/выкл пользователя (обновить БД + VPS API)
   - `DELETE /api/admin/users/{id}` — удалить (БД + VPS API)
   - `GET /api/config/{token}` — получить данные для персональной страницы (public, без auth)
   - `POST /api/admin/sync` — полная синхронизация UUID из БД → VPS
   - `GET /api/cron/stats` — cron endpoint (защищён `CRON_SECRET` в заголовке `Authorization: Bearer {CRON_SECRET}`)
4. Middleware: проверка JWT cookie `veilx-admin` для /api/admin/* routes. Невалидный/просроченный токен → 401 JSON `{ error: "Unauthorized" }`

### Фаза 5: Страницы (Frontend)

**Зависимости**: Фаза 4 (нужны API routes).

1. Главная (`/`):
   - Статический лендинг с Tailwind
   - Кнопка «Инструкция» → `/setup`
2. Инструкции (`/setup`):
   - Табы по платформам: iOS, Android, Windows, macOS
   - Ссылки на скачивание клиентов
   - Пошаговые инструкции (текст, без скриншотов в MVP)
3. Персональная страница (`/c/[token]`):
   - SSR: запрос к БД по token → данные пользователя
   - Формирование vless:// ссылки на сервере
   - QR-код (генерация на клиенте из vless:// ссылки)
   - Кнопка «Скопировать» (Clipboard API)
   - Обработка состояний: active, disabled, not found
4. Админ-панель (`/admin`):
   - Форма входа (пароль)
   - Форма создания инвайта (имя → POST → показать ссылку)
   - Таблица пользователей с действиями
   - Баннер ошибки связи с VPS

### Фаза 6: Cron и мониторинг

**Зависимости**: Фазы 2 + 4 (VPS API + API routes).

1. Настроить Vercel Cron в `vercel.json`:
   ```json
   { "crons": [{ "path": "/api/cron/stats", "schedule": "0 3 * * *" }] }
   ```
   Расписание: раз в сутки (03:00 UTC). Hobby plan не поддерживает hourly cron.
   Endpoint защищён `CRON_SECRET` — Vercel автоматически передаёт его для cron-вызовов.
2. Реализовать сбор статистики: Vercel Cron → VPS API `/stats` → обновить Postgres
3. Проверить работу: вручную вызвать `/api/cron/stats`, убедиться что данные обновляются

### Параллельность

```
Фаза 1 (VPS) ──→ Фаза 2 (VPS API) ──┐
                                       ├──→ Фаза 4 (API routes) → Фаза 5 (Frontend) → Фаза 6 (Cron)
Фаза 3 (БД)  ─────────────────────────┘
```

Фазы 1 и 3 можно делать параллельно. Фазы 2 и 3 должны завершиться до начала Фазы 4.

## 10. ДОПОЛНИТЕЛЬНЫЕ РЕШЕНИЯ

### CORS
- VPS API: CORS не нужен — вызывается только из Vercel API routes (server-to-server).
- Next.js API routes: CORS не нужен — фронтенд и API на одном домене (veilx.app).

### Бэкап БД
- Vercel Postgres делает автоматические бэкапы (managed service).
- Ручной экспорт: `pg_dump` через connection string раз в месяц (опционально).

### Обновление Xray
- Проверять releases на GitHub раз в месяц.
- Обновление: скачать бинарник → заменить → `systemctl restart xray`.
- Перед обновлением: `cp /usr/local/etc/xray/config.json /usr/local/etc/xray/config.json.bak`.

### Восстановление пароля админа
- Пароль хранится в env на Vercel. При потере — изменить через Vercel Dashboard → Environment Variables → Redeploy.

### Аудит-лог (post-MVP)
- Логировать действия админа (создание/удаление/отключение юзеров) в таблицу `admin_log`.
- В MVP достаточно `console.log` в API routes — логи видны в Vercel Dashboard.
