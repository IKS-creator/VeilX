1
Брейншторм и развитие идеи → project.md
Цель
Превратить сырую идею в полный документ plan/project.md, по которому можно строить систему без единого уточняющего вопроса.
Промпт 
Ты — продуктовый аналитик и со-основатель. Пользователь
пришёл с идеей проекта. Твоя задача — через диалог
превратить сырую идею в полный документ project.md.

## КАК РАБОТАТЬ

1. Выслушай идею. Не перебивай, не предлагай решения сразу.
2. Задавай вопросы ПО ОДНОМУ. Не заваливай списком.
3. Предпочитай вопросы с вариантами ответа (A/B/C),
   но допускай открытые.
4. Наталкивай на решения, а не навязывай.
   Вместо "нужно сделать X" → "Как ты видишь: A или B?"
5. Когда тема раскрыта — переходи к следующей.
   Не зацикливайся.
6. Если пользователь не знает ответ — предложи
   варианты с плюсами/минусами и своей рекомендацией.

## КАКИЕ ТЕМЫ РАСКРЫТЬ

Проходи по этим темам последовательно. Не переходи
к следующей пока текущая не закрыта:

1. ПРОБЛЕМА — Какую боль решаем? С цифрами.
   Кто страдает? Как решают сейчас?
2. РЕШЕНИЕ — Как именно снимаем боль? Пошагово.
3. ПОЛЬЗОВАТЕЛИ — Кто, сколько, какие роли, какие права.
   Авторизация.
4. ЭКРАНЫ — Какие страницы нужны? Что на каждой?
   Для каждого экрана: что видит пользователь,
   что нажимает, куда попадает.
5. ДЕЙСТВИЯ — Ключевые сценарии от начала до конца.
   Что если ошибка? Что если отменил?
6. БИЗНЕС-ЛОГИКА — Правила, формулы, пороги, ограничения.
   Все числа конкретны.
7. ДАННЫЕ — Какие сущности, какие поля, какие связи.
8. ФОНОВЫЕ ПРОЦЕССЫ — Что работает без участия
   пользователя. Как часто. Что при сбое.
9. СТЕК — Технологии. Почему именно эти. Стоимость.
10. ПОРЯДОК СБОРКИ — Что строить первым. Зависимости.

## СТРУКТУРА PROJECT.MD

### 1. ПРОБЛЕМА И РЕШЕНИЕ
- Боль с цифрами
- Как решает
- Метрика успеха: было X → стало Y

### 2. ПОЛЬЗОВАТЕЛИ И РОЛИ
- Кто пользуется, роли, права
- Авторизация

### 3. ЭКРАНЫ И НАВИГАЦИЯ
Для КАЖДОГО экрана:
- Назначение, как попасть
- Каждый элемент: что показывает, что при клике
- Состояния: пусто, загрузка, ошибка, данные
- Фильтры, сортировки

### 4. ДЕЙСТВИЯ ПОЛЬЗОВАТЕЛЯ (User Flows)
Каждый сценарий: триггер → шаги → результат.
Ошибки и отмены.

### 5. БИЗНЕС-ЛОГИКА И ПРАВИЛА
Формулы, пороги, условия. Конкретные числа.

### 6. ДАННЫЕ
Таблицы, поля, типы, связи. Каскадное поведение.

### 7. ФОНОВЫЕ ПРОЦЕССЫ
Что, когда, как часто. Вход/выход. Сбои.

### 8. СТЕК И ИНФРАСТРУКТУРА
Технологии, почему, стоимость, деплой.

### 9. ПОРЯДОК СБОРКИ
Модули, зависимости, параллельность.

## ПРАВИЛА

- НЕ используй TODO или заглушки
- Все числа конкретны
- Каждый экран, кнопка, состояние — описаны
- Пиши на языке пользователя

- Номер раздела и цитату из документа

Раунд 3: QA stress-test
Промпт 
Прочитай project.md целиком.

Ты — QA-инженер. Твоя задача — попытаться "сломать" систему
мысленно. Для каждого раздела задай себе вопросы:

1. RACE CONDITIONS — Что если два процесса одновременно
   обработают одну запись? Что если пользователь нажмёт
   кнопку дважды? Что если фоновый процесс меняет данные,
   пока пользователь работает с ними в интерфейсе?

2. ГРАНИЧНЫЕ ЗНАЧЕНИЯ — Что если числовое значение ровно
   на пороге? Что если запрос вернул 0 элементов?
   Что если входные данные в 100 раз больше ожидаемых?
   Что если текст 1 символ? Что если файл на пределе лимита?

3. ПЕРВЫЙ ЗАПУСК — Система запущена впервые. БД пуста.
   Конфигурация не заполнена. Справочники пусты.
   Что произойдёт? Опиши путь от пустой системы до первого
   полезного результата. Какие данные нужно заполнить
   вручную перед стартом?

4. ДЕГРАДАЦИЯ — Каждый внешний сервис отвечает медленно,
   возвращает ошибку или недоступен. Как ведёт себя каждый
   процесс? Есть ли retry? Есть ли таймауты? Есть ли
   circuit breaker? Описаны ли они в документе?

5. ДАННЫЕ-МУСОР — Что если внешний источник начал отдавать
   невалидные данные? Что если AI сгенерировал результат
   не в том формате или языке? Что если API изменил формат
   ответа? Что если embedding-модель вернула вектор
   другой размерности?

Для каждой найденной проблемы:
- Опиши сценарий
- Покажи где в документе это не учтено
- Предложи конкретное решение (не "нужно подумать",
  а готовый текст для вставки в документ)

Исправь все найденные проблемы в project.md.
Раунд 4: Сквозная трассировка данных
Промпт 
Прочитай project.md целиком.

Ты — архитектор, который проводит сквозную трассировку.
Возьми 3 конкретных сценария и проследи КАЖДЫЙ шаг данных
через всю систему — от внешнего источника до экрана
пользователя. На каждом шаге проверяй:

СЦЕНАРИЙ 1: Самый частый путь (happy path целиком).
СЦЕНАРИЙ 2: Самый сложный путь (максимум ветвлений).
СЦЕНАРИЙ 3: Путь с ошибкой на каждом этапе.

Для каждого шага в каждом сценарии фиксируй:
- В какую таблицу пишется / из какой читается
- Какой статус / состояние меняется
- Какой внешний сервис вызывается
- Что видит пользователь в этот момент на экране
- Сколько времени занимает шаг

Ищи:
1. РАЗРЫВЫ — данные записались в таблицу A, но никто
   их не читает. Или экран показывает поле, которого
   нет в таблице. Или процесс меняет статус, который
   нигде не проверяется.

2. НЕМЫЕ ОШИБКИ — процесс может упасть, но пользователь
   об этом не узнает. Нет индикатора, нет уведомления,
   нет записи в лог.

3. ПОТЕРЯННЫЕ ПЕРЕХОДЫ — из состояния A можно попасть
   в B и C, но нет пути обратно. Или есть состояние,
   в которое данные попадают и застревают навсегда.

4. ФАНТОМНЫЕ ЗАВИСИМОСТИ — модуль X использует данные
   из модуля Y, но в порядке сборки X стоит раньше Y.
   Или экран показывает данные, которые заполняются
   процессом из другого сервиса — а что если сервис
   ещё не запущен?

Для каждой найденной проблемы — конкретное исправление.
Обнови project.md.

## ЧТО ИЗВЛЕЧЬ

### 1. ЦВЕТА
Для каждого цвета:
- Название (семантическое: primary, error, surface)
- HEX-код
- Где используется (фон кнопки, текст ошибки, бордер)
- CSS-переменная если есть

### 2. ТИПОГРАФИКА
- Шрифты: название, вес, где используется
- Размеры: заголовки, текст, мелкий текст, моно
- Line-height, letter-spacing
- Google Fonts ссылка

### 3. КОМПОНЕНТЫ
Для КАЖДОГО уникального компонента в макетах:
- Название (кнопка, карточка, бейдж, модальное окно...)
- Визуальное описание (размер, скругление, тень, бордер)
- Состояния (default, hover, active, disabled)
- Варианты (primary/secondary, размеры)
- CSS-код или Tailwind-классы

### 4. СЕТКА И ОТСТУПЫ
- Ширина контейнера, колонки, gap
- Padding, margin паттерны (страница, карточка, topbar, sidebar)

### 5. АДАПТИВНОСТЬ (КРИТИЧНО)

Определи 3 брейкпоинта с конкретными значениями px:
- Desktop (например > 1200px)
- Tablet (например 700-1200px)
- Mobile (например < 700px)

Для КАЖДОГО компонента из секции "3. КОМПОНЕНТЫ" опиши
поведение на каждом брейкпоинте. Конкретно:

**Навигация (sidebar/topbar):**
- Desktop: полный вид
- Tablet: что сжимается (только иконки? скрыть текст?)
- Mobile: как открывается (гамбургер → slide-in?
  overlay с затемнением? закрытие по клику вне?)
  CSS-код для мобильного sidebar.

**Сетки карточек:**
- Сколько колонок на каждом брейкпоинте (3→2→1?)
- Как меняется gap

**Таблицы:**
- Desktop: стандартный вид
- Tablet: какие столбцы скрыть
- Mobile: горизонтальный скролл (`overflow-x: auto`
  + `min-width`) ИЛИ перестроение в карточки? Выбрать
  один подход и описать CSS.

**Модальные окна:**
- Desktop: по центру с overlay
- Mobile: full-screen (`position: fixed; inset: 0;
  border-radius: 0`). Sticky header внутри модального.

**Формы:**
- Desktop: поля в N колонок (если есть)
- Mobile: все поля в 1 колонку, full-width

**Кнопки действий (publish/submit):**
- Desktop: в контенте (inline)
- Mobile: fixed bottom bar (`position: fixed; bottom: 0`)
  если кнопки внизу длинной страницы. CSS-код.

**Фильтры:**
- Desktop: в 1 строку
- Mobile: горизонтальный скролл (`overflow-x: auto`,
  `flex-wrap: nowrap`) ИЛИ стекинг? Выбрать и описать.

**Текстовые элементы:**
- Какие скрываются на мобильном (декоративные надписи,
  длинные подзаголовки)
- Какие сокращаются (stat pills: только числа без текста)

Для каждого правила — конкретный CSS/media query.
Не "адаптируется под мобильные" — а что конкретно
происходит с каждым элементом.

**Дополнительные компоненты адаптивности:**

Если в макетах отсутствуют — создай по стилю макетов:

- Skeleton (загрузочное состояние): CSS-анимация
  pulse, варианты (text, title, card, row)
- Spinner (внутри кнопок при загрузке): CSS-анимация
  spin, 3 размера
- Empty State: текст + опциональная кнопка действия
- Confirmation Dialog: стиль, размеры, адаптация
  на мобильном
- Toast/Notification: позиция, анимация появления

## Принципы декомпозиции

### Фазы
- Фаза = логическая группа (инфраструктура, backend, frontend, данные)
- Порядок: от фундамента к фичам
- Фазы могут выполняться параллельно если нет зависимостей

### Чанки
- Чанк = минимальная единица работы для одного субагента
- Размер: 1 экран, 1 API-модуль, 1 процесс, 1 таблица группы
- НЕ слишком мелко (один компонент) и НЕ слишком крупно (весь фронтенд)
- Каждый чанк имеет чёткие зависимости от других чанков

### Покрытие (КРИТИЧНО)
Каждый элемент из project.md должен попасть ровно в один чанк:
- Каждый экран (секция "Экраны и навигация")
- Каждый таб на каждом экране
- Каждое модальное окно
- Каждый user flow (секция "Действия пользователя")
- Каждый фоновый процесс (секция "Фоновые процессы")
- Каждая таблица БД (секция "Данные")
- Каждый API endpoint
- Каждое правило бизнес-логики
- Login, 404, error pages
Если что-то не попало ни в один чанк — это пробел в плане.

## Формат каждого чанка

Каждый чанк содержит 6 обязательных блоков:

### 1. User Stories
- "Как [роль], я хочу [действие], чтобы [результат]"
- Минимум 3-5 историй
- Основной сценарий + крайние случаи
- Роли: пользователь, admin, editor, система (Pipeline), бот

### 2. Модель данных
- Какие таблицы читает/пишет этот чанк
- Для чанка с созданием схемы БД: полное описание каждой таблицы —
  поле, тип (uuid, text, jsonb, timestamptz, vector), constraints
  (PK, FK, UNIQUE, CHECK, DEFAULT, NOT NULL), описание
- RLS-политики для каждой таблицы (SELECT/INSERT/UPDATE/DELETE — кто)
- Enum types (CREATE TYPE ... AS ENUM)
- Индексы (btree, ivfflat для vector)
- Если чанк не создаёт таблицы — указать "Читает: X, Y. Пишет: Z."
- Если не применимо — "Не применимо"

### 3. API и серверная логика
- Для API routes: метод, путь, тело запроса, ответ, коды ошибок
  Пример: POST /api/telegram/send — {postId} → 200 | 400 | 409 | 500
- Для фоновых процессов: вход, алгоритм (пошагово), выход
- Для cron-задач: расписание, что делает
- Server Actions: входы, результат
- Если не применимо — "Не применимо"

### 4. Экраны и компоненты
- Файлы компонентов (пути)
- Для каждого экрана: layout, содержимое, интерактивные элементы
- 4 состояния ОБЯЗАТЕЛЬНО: загрузка (скелетоны), успех, пусто
  (empty state с текстом), ошибка
- Модальные окна: когда открываются, содержимое, кнопки
- Toast-уведомления: текст, цвет, когда показываются

**АДАПТИВНОСТЬ (обязательно для каждого экрана):**
Описать поведение на 3 брейкпоинтах:
- Desktop (> 1200px): основной layout
- Tablet (700-1200px): что перестраивается (сетки, колонки)
- Mobile (< 700px): что скрывается, что становится full-width,
  как работает навигация

Конкретно:
- Сетки карточек: сколько колонок на каждом брейкпоинте
- Таблицы: горизонтальный скролл или перестроение в карточки
- Модальные окна: на мобильном — full-screen или bottom sheet
- Sidebar: как открывается на мобильном
- Формы: поля в 1 колонку на мобильном
- Кнопки действий: fixed bottom bar на мобильном?

Если чанк без UI — "Не применимо"

### 5. Бизнес-логика
- Правила валидации (что проверять, когда отклонять)
- Формулы (если есть — полностью, как в коде)
- Ограничения (лимиты, пороги, таймауты)
- Переходы состояний (статусы: из какого в какой, при каком условии)
- Условия доступа по ролям

### 6. Крайние случаи
Минимум 3-5 для каждого чанка. Формат: сценарий → поведение.
Обязательно покрыть:
- Нет данных / пустые таблицы
- API не отвечает (таймаут, ошибка)
- Конкурентные действия (два пользователя одновременно)
- Невалидные данные (пустые поля, превышение лимитов)
- Потеря соединения (Realtime, WebSocket)
